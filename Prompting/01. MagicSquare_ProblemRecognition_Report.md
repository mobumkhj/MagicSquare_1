# STEP 1 — Observation (관찰)

워크스페이스(`MagicSquare_xx`)에는 아직 코드나 문서가 없습니다. 즉, **“이미 돌아가는 프로그램을 고치는 상황”이 아니라, 무엇을 만들지 정하기 전의 출발점**에 있습니다. 아래는 “마방진을 만든다”는 목표 문장 대신, **관찰자가 보는 상황**으로 씁니다.

---

## 1. 현재 우리가 해결하려는 상황은 무엇인가?

**한 사람이 4×4 격자에 1부터 16까지의 정수를 한 번씩 배치했을 때, 여러 줄·열·대각선의 합이 모두 같아지는 배치를 찾거나 검증·생성하는 과정을, 컴퓨터 프로그램으로 다루고 싶어 하는 상황**입니다.

관찰하면 이런 요소가 보입니다.

| 관찰 | 의미 |
|------|------|
| 격자 크기가 4×4로 고정 | “임의 크기”가 아니라 **제약이 명확한 소규모 조합 문제** |
| 사용 숫자가 1~16, 중복 없음 | **순열·배치** 문제이면서 동시에 **산술 제약(합 동일)** 문제 |
| 행·열·대각선 합이 같음 | “맞는 배치”의 정의가 **규칙으로 명시**되어 있음 |
| 프로그램으로 다룸 | 사람이 손으로 채우는 것이 아니라 **반복·탐색·검증을 기계에 맡기려는 의도** |

정리하면, 상황은 **“16칸 배치 + 합 일치라는 여러 조건을 동시에 만족하는 상태를, 어떻게 찾고 어떻게 보여줄 것인가”**에 가깝습니다. 아직은 구현 방법(알고리즘, UI, 입출력 형식)이 정해지지 않은 **문제 정의 직전** 단계입니다.

---

## 2. 왜 4×4 마방진 문제를 다루는가?

“4×4만”이 아니라, **이 크기와 이 규칙이 가진 성질** 때문에 자주 선택됩니다.

**규모**

- 16칸, 1~16 → 전수·탐색·검증을 **손계산·종이·디버깅**으로도 따라갈 수 있는 크기  
- 3×3보다 제약이 풍부하고, 5×5 이상보다 **복잡도가 한 단계 올라가는 데 적당**함  

**수학·규칙**

- 4×4는 **짝수 차수** 마방진의 대표 사례(홀수 차수와 다른 성질·구성 방법이 논의됨)  
- “모든 줄의 합이 같다”는 조건이 **한 줄의 목표 합(마법 상수)**으로 수식화되기 쉬움  

**학습·실습**

- 조건 검사, 배열/2차원 구조, 반복, (필요 시) 백트래킹·휴리스틱 등 **프로그래밍 기본기**를 한 문제에 묶기 좋음  
- “맞는지 검증”과 “하나를 만들기”를 **역할을 나눠** 설계 연습하기 좋음  

**시스템 관점**

- 입력(크기·규칙·시드) → 처리(생성/탐색) → 출력(격자·합 검증 결과) 같은 **작은 파이프라인**을 연습하기에 부담이 적음  

즉, “4×4 마방진”은 목적 그 자체라기보다, **제약이 분명하고, 검증 가능하고, 확장(5×5, 일반 n)으로 이어지기 쉬운 표준 연습 문제**로 자주 쓰입니다.

---

## 3. 어떤 학습 또는 시스템 설계 맥락에서 등장하는가?

**학습 맥락**

- **알고리즘·자료구조**: 2차원 배열, 인덱스, 행/열/대각선 순회, 조건 만족 여부 판별  
- **문제 해결 절차**: 관찰 → 정의 → 모델링 → 알고리즘 선택(이 단계는 지금 여기)  
- **수학·논리**: 불변량(각 줄의 합 = 마법 상수), 대칭·패턴, “해가 여러 개인지/유일한지”  
- **소프트웨어 공학 입문**: 요구사항(“합이 같은 4×4”), 테스트(주어진 격자가 유효한지), 모듈(생성 vs 검증)  

**시스템 설계 맥락**

- **도메인 규칙 엔진에 가까운 작은 코어**: “유효한 4×4 배치”의 정의가 곧 비즈니스 규칙  
- **검증 서비스 vs 생성 서비스**: 같은 규칙을 공유하는 두 기능(읽기/쓰기 분리 연습)  
- **확장 가능성의 씨앗**: 지금은 4×4 고정이지만, 나중에 n×n, 부분 마방진, 사용자 입력 격자 검증 등으로 **경계를 넓히는 연습**  
- **관찰 가능성**: 각 줄 합, 마법 상수, 실패한 줄 번호 등을 **로그/출력으로 드러내는** 설계  

**상황을 한 문장으로 (관찰 관점)**

> 학습자 또는 개발자가, **고정된 격자와 숫자 집합 위에서 “여러 방향의 합이 같다”는 규칙을 만족하는 배치**를 다루기 위해, 그 규칙을 프로그램이 이해·검증·(필요 시) 탐색할 수 있게 만들려는 **초기 문제 인식** 단계에 있다.

# STEP 2 — Why #1

## 가정한 답 (A)

> **“4×4 격자에 1~16을 한 번씩 넣었을 때, 모든 행·열·두 대각선의 합이 같아지는 ‘완성된 배치’가 있어야, 그 배치가 마방진 규칙을 만족한다는 것을 확인할 수 있고, 프로그램의 목적(올바른 결과 제공)을 달성했다고 말할 수 있기 때문이다.”**

즉, **완성 = 규칙 충족의 증거**이고, **미완성 = 성공 여부를 판단할 수 없음**이라는 전제를 둡니다.

---

## A에서 드러나는 불편함·구조적 문제

### 1. “완성”과 “목적”이 한 가지로 묶여 있음

A는 **완성된 격자**를 거의 유일한 성공 기준으로 둡니다. 그러면 다음이 애매해집니다.

| 상황 | A만 따르면 |
|------|-----------|
| 사용자가 칸을 직접 채우며 학습 | “아직 완성 안 됨” = 실패처럼 보임 |
| 프로그램이 **검증만** 하면 됨 | “완성”이 필수 목적인지 불명확 |
| 이미 알려진 해를 **출력만** 하면 됨 | “만들기”와 “보여주기”가 같은 이유인지 구분 안 됨 |

**구조적 문제:** *왜 완성해야 하는가*에 대한 답이 **생성·검증·학습·시연**을 한 문장에 합쳐 버려, 나중에 기능을 나누기 어렵습니다.

---

### 2. “완성”의 정의가 암묵적임

A는 “합이 모두 같다”는 **최종 상태**를 말하지만, 다음은 말하지 않습니다.

- **부분 배치**(일부 칸만 채움)는 어디까지 허용하는가?
- **행만** 맞고 대각선은 아직인 경우는 “진행 중”인가 “오류”인가?
- **해가 여러 개**일 때, 하나만 내면 “완성”인가, 전부 나열해야 “완성”인가?

**불편함:** 구현·테스트 시 “완성됐다”의 판정 기준이 흔들리고, **중간 상태를 다루는 요구**가 뒤늦게 튀어나옵니다.

---

### 3. 완성이 곧 가치인지, 수단인지 불분명

A의 논리: 완성 → 규칙 만족 확인 → 목적 달성.

그런데 실제 맥락에서는:

- **학습:** 과정(왜 이 칸에 이 숫자인지)이 가치일 수 있음 → 완성은 **부산물**
- **알고리즘 실습:** 탐색·백트래킹 **과정**이 가치 → 완성은 **검증용 결과**
- **제품:** “유효한 마방진인지 검사”만 필요 → 완성 **생성**은 필수가 아님

**구조적 문제:** “왜 완성해야 하는가”에 **가치(학습/신뢰/재미)**가 빠져 있어, 요구가 바뀔 때(검증만, 힌트만) **첫 번째 Why가 무너진 것처럼** 느껴집니다.

---

### 4. 실패·불가능에 대한 자리가 없음

4×4 + 1~16 + 표준 마방진 규칙이면 **해는 존재**하지만, A는 “완성해야 한다”만 강조합니다.

- 탐색이 **오래 걸리거나** 잘못된 알고리즘이면?
- 사용자가 **불가능한 부분 배치**를 넣으면?
- “완성 못 함”을 **언제·어떻게** 알려줄지?

**불편함:** 성공 경로(완성)만 정의되어 **미완성·불가능·타임아웃**이 시스템 설계에서 뒤처집니다.

---

### 5. 사람의 “완성”과 기계의 “완성”이 충돌할 수 있음

| 관점 | “완성”의 의미 |
|------|----------------|
| 사람 | 눈으로 대칭·패턴이 맞아 보임, 또는 퍼즐 규칙 만족 |
| 프로그램 | 모든 줄 합이 동일, 숫자 1~16 각 1회 |

**구조적 문제:** A는 **수학적 완성**만 전제합니다. UI·메시지·진행 표시를 설계할 때 **“거의 맞음”(한 줄만 틀림)** 같은 상태가 설계 밖으로 밀립니다.

---

### 6. 검증 가능성과 생성 부담이 뒤섞임

A의 핵심은 사실 **“규칙을 만족함을 확인”**에 가깝습니다.  
그런데 “완성해야 한다”는 표현은 **반드시 새로 만들어야 한다**는 뉘앙스를 줍니다.

- **검증만** 하면: 입력 격자 → 합·중복 검사 → 완성 여부 판정 (상대적으로 단순)
- **생성까지** 하면: 탐색·구성 규칙·성능·해의 개수 (훨씬 무거움)

**불편함:** Why #1이 **생성 의무**까지 끌고 가면, STEP 1에서 본 “작은 연습 문제”가 **불필요하게 큰 시스템**으로 커질 위험이 있습니다.

# STEP 3 — Why #2

**Q: 왜 단순 계산이 아니라 프로그램으로 구현하는가?**

4×4, 1~16, 행·열·대각선 합 동일 — 이 규모는 종이·계산기로도 “한 번”은 따라갈 수 있습니다. 그럼에도 **프로그램**을 선택하는 이유를 네 가지 관점에서 정리합니다. (아직 설계·구현은 하지 않습니다.)

---

## 1. 반복 가능성 (Repeatability)

**관찰:** 손 계산은 같은 사람도, 같은 문제도 **실행마다 속도·순서·기록 방식**이 달라질 수 있습니다. 프로그램은 **같은 입력·같은 규칙**이면 **같은 절차**를 다시 실행할 수 있습니다.

| 손 계산 | 프로그램 |
|--------|----------|
| 한 번 풀고 끝나기 쉬움 | N번, 다른 시드·다른 초기 격자에도 동일 절차 적용 |
| “다시 해보기”가 피로도와 함께 증가 | 반복 실행 비용이 거의 일정 |
| 과정이 머릿속·메모에만 남음 | 과정·결과를 **재현 가능한 형태**(로그, 출력)로 남기기 쉬움 |

**Why에 대한 답:**  
마방진을 **한 번 맞추는 것**이 아니라, **같은 규칙으로 여러 번·여러 경우**를 다루려면, 사람의 반복보다 **기계적 반복**이 구조적으로 유리합니다.  
(예: 다른 시작 칸, 여러 해 탐색, 수업에서 학생마다 다른 입력 검사 등)

**드러나는 긴장:**  
“반복”이 목표면, **무엇을 반복할지**(생성만 / 검증만 / 둘 다)가 아직 Why #1에서처럼 분리되지 않으면, 프로그램 범위가 불필요하게 커질 수 있습니다.

---

## 2. 검증 자동화 (Automated Verification)

**관찰:** “맞다”의 기준은 **규칙의 나열**입니다. (16개 숫자 각 1회, 4행·4열·2대각선 합 일치 등.) 사람은 눈으로 훑으며 확인하지만, **항목이 많을수록 빠뜨리기 쉽습니다.**

**프로그램이 하는 일(개념 수준):**

- 격자를 읽고 → 행/열/대각선 합 계산 → 목표 합(마법 상수)과 비교  
- 1~16 사용 여부·중복 검사  
- 실패 시 **어느 줄이 틀렸는지**까지 기계적으로 보고 가능  

**Why에 대한 답:**  
완성 여부를 **주관적 확신**이 아니라 **규칙 목록에 대한 일괄 판정**으로 바꾸려면, 검증을 **자동화**하는 쪽이 자연스럽습니다.  
손 계산은 “대략 맞는 것 같다”에서 멈출 수 있고, 프로그램은 **정의된 검사 전부**를 매번 동일하게 적용합니다.

**드러나는 긴장:**  
Why #1에서 “완성”이 목표였다면, Why #2에서는 실제 핵심이 **“완성을 만드는 것”보다 “완성 여부를 기계적으로 증명하는 것”**일 수 있습니다.  
→ **생성 없이 검증 모듈만**으로도 이 Why는 상당 부분 충족됩니다.

---

## 3. 오류 방지 (Error Prevention)

**관찰:** 손 계산·머리 계산에서 흔한 오류 유형:

- 한 숫자 **중복·누락**  
- 한 줄만 맞고 **다른 줄·대각선** 미확인  
- 합 계산 **산술 실수**  
- 패턴 복사 시 **인덱스 한 칸 밀림**  

4×4는 작아도 **확인해야 할 제약이 많습니다**(행 4 + 열 4 + 대각선 2 + 숫자 집합 1).

**프로그램이 기대하는 효과:**

| 오류 유형 | 프로그램 쪽 대응(개념) |
|----------|---------------------|
| 산술 실수 | 합·비교를 코드에 위임 |
| 검사 누락 | 검증 단계를 빠짐없이 실행 |
| 규칙 해석 흔들림 | “유효함”의 정의를 코드·테스트로 고정 |

**Why에 대한 답:**  
**사람의 주의력**에 의존하는 대신, **규칙을 한곳에 모아 두고 매번 전부 적용**함으로써, 실수 가능 지점을 줄이려는 선택입니다.  
특히 “거의 맞은” 격자에서 **한 줄만 틀린 경우**를 사람이 놓치기 쉬운데, 자동 검증은 그 차이를 드러내기 쉽습니다.

**드러나는 긴장:**  
프로그램도 **버그**는 낼 수 있습니다. 오류 방지의 대상이 **사용자(사람)의 계산 실수**인지, **개발자(구현)의 논리 오류**인지를 구분하지 않으면, “프로그램이면 안전하다”는 과신이 생깁니다.  
→ **테스트·검증 로직 자체의 정확성**도 별도 Why/요구가 됩니다.

---

## 4. 규칙 기반 사고 훈련 (Rule-Based Thinking Practice)

**관찰:** 마방진은 숫자 놀이이면서 동시에 **명시적 규칙의 집합**입니다.  
“이 칸에 7을 넣을 수 있다”는 **감**이 아니라, **아직 쓰이지 않은 수 + 넣었을 때 어떤 줄 합이 깨지는지** 같은 규칙으로 말할 수 있습니다.

**프로그램 구현이 훈련에 주는 것(학습 맥락):**

- 자연어 규칙 → **조건·데이터 구조·함수**로 옮기는 연습  
- “유효한 배치”를 **판정 함수**로 표현  
- “다음 후보”를 **탐색·제약**으로 표현 (생성을 할 경우)  
- 규칙 변경 시(예: 대각선 제외) **어디를 고쳐야 하는지** 보이는 구조  

**Why에 대한 답:**  
단순 계산은 **결과 하나**에 머무르기 쉽고, 프로그램은 **규칙을 실행 가능한 형태로 고정**해야 하므로, **규칙을 분해·조합·검증하는 사고**를 훈련하기에 적합합니다.  
즉, 목적이 “16칸을 채운다”만이 아니라 **“규칙을 시스템으로 다루는 법을 익힌다”**일 때 프로그램이 의미가 큽니다.

**드러나는 긴장:**  
규칙 기반 사고만 목표라면 **과도한 기능**(모든 해 나열, 화려한 UI)은 오히려 산만해질 수 있습니다.  
→ **최소 규칙 세트 + 검증 + (선택) 생성** 정도가 학습 목적과 잘 맞는지, 다음 단계(문제 정의)에서 맞춰야 합니다.

---

## 네 관점을 한 줄로 묶은 Why #2

> **4×4 마방진을 프로그램으로 다루는 이유는, 같은 규칙을 여러 번·여러 입력에 재현하고(반복), “맞음”을 사람의 눈 대신 규칙 전체로 판정하며(검증 자동화), 누락·산술·부분 확인 같은 실수를 줄이고(오류 방지), 규칙을 코드 수준에서 명시·실행 가능하게 만드는 훈련(규칙 기반 사고)을 하기 위해서이다.**

# STEP 4 — Why #3 (핵심 발견)

**Q: 왜 이 문제를 TDD 방식으로 설계하려 하는가?**

STEP 1~3에서 이미 보였습니다. **규칙이 명시적**이고, **맞음/틀림이 판정 가능**하며, **프로그램의 가치가 “검증·반복·오류 방지”**에 가깝습니다. TDD는 이런 문제에서 **“무엇이 맞는지”를 코드보다 먼저 고정**하는 설계 방식입니다. 아래 세 질문에 답합니다.

---

## 1. 무엇이 통제되어야 하는가?

TDD로 두려는 것은 **구현의 자유도**가 아니라, **도메인 규칙이 항상 지켜지는지**입니다.

| 통제 대상 | 왜 통제하는가 | TDD에서의 형태(개념) |
|-----------|----------------|----------------------|
| **유효성 판정** | “마방진인가?”의 기준이 흔들리면 성공/실패를 말할 수 없음 | 검증 함수에 대한 테스트 |
| **숫자 집합** | 1~16 각 1회가 깨지면 합이 맞아도 무효 | 중복·누락·범위 테스트 |
| **합 일치** | 행·열·대각선 각각이 목표 합과 같아야 함 | 줄별 합, 마법 상수 테스트 |
| **경계·형태** | 4×4가 아니면 문제 정의 자체가 달라짐 | 크기·빈 칸·잘못된 입력 테스트 |
| **생성 vs 검증** | Why #1~2에서 섞였던 책임 | 역할별 테스트 스위트 분리 |
| **회귀** | 알고리즘·리팩터 후 “예전에 맞던 격자”가 깨지지 않음 | 고정된 예시 격자(골든 케이스) |

**핵심 발견:**  
이 문제에서 통제해야 할 것은 **UI나 출력 예쁨이 아니라 “규칙 엔진의 판정 결과”**입니다. TDD는 **“개발자가 느낌으로 맞다고 한 것”**을 **실행 가능한 기대값**으로 바꿔 통제합니다.

**Why #2와의 연결:**  
검증 자동화·오류 방지를 목표로 했다면, **그 검증 로직 자체가 틀리면 안 됩니다.** 통제의 1순위는 **검증의 정확성**이고, 그다음이 생성·탐색입니다.

---

## 2. 어떤 불변 조건이 존재하는가?

**불변 조건(invariant)** 은 상태가 바뀌어도(칸을 채우거나, 알고리즘 단계가 진행되어도) **항상 참이어야 하는 성질**입니다. 4×4 마방진에서는 수학·조합적으로 여러 층이 있습니다.

### (A) 완성된 격자에서의 불변 조건 (최종 상태)

| 불변 조건 | 내용 |
|-----------|------|
| **집합 불변** | 16칸의 값 집합 = {1, 2, …, 16} (각 정확히 1회) |
| **마법 상수** | 모든 행·열·두 대각선의 합 = 동일한 값 M |
| **M의 값** | 4×4, 1~16일 때 M = 34 (미리 계산 가능한 불변량) |
| **대칭·구조(선택)** | 특정 구성법(Siamese, Strachey 등)을 쓰면 추가 패턴 불변 — **요구에 넣을 때만** 테스트 대상 |

→ TDD에서는 **“완성 격자 G는 항상 `isValid(G) == true`”** 가 가장 먼저 고정되는 불변입니다.

### (B) 생성·탐색 중간 상태에서의 불변 조건 (과정)

| 불변 조건 | 내용 |
|-----------|------|
| **부분 집합** | 이미 넣은 수는 서로 다르고, 1~16 안에 있음 |
| **미사용 수** | 아직 안 넣은 수만 후보 |
| **부분 검증(약한 불변)** | “지금까지 채운 줄만” 합을 검사 — **너무 이른 가지치기**에 쓰는 휴리스틱이지, 최종 불변과 동일하지 않음 |

**핵심 발견:**  
**최종 불변**(완성 시 M=34, 전 줄 합 일치)과 **중간 불변**(부분 배치)을 **테스트에서 구분**하지 않으면, “아직 대각선 안 채웠는데 실패” 같은 **거짓 실패**를 설계에 넣게 됩니다.

### (C) 프로그램·모듈 경계에서의 불변 조건

| 불변 조건 | 내용 |
|-----------|------|
| **검증기 단조성** | 같은 격자 입력 → 같은 `true/false` (순수 함수) |
| **생성기 계약** | `generate()`가 반환한 격자는 **항상** `validate()` 통과 (생성을 할 경우) |
| **실패 명시** | 불가능/미완성 입력 → 예외 또는 `false` + 이유 — **침묵 실패 금지** (Why #1의 “미완성” 문제 대응) |

TDD는 이런 불변을 **테스트 이름·assert 문장**으로 먼저 적어, 구현이 **불변을 깨는 순간** 빨간불이 켜지게 합니다.

## 3. 왜 입력/출력이 명확해야 하는가?

TDD의 한 사이클은 **“주어진 입력 → 기대 출력 → 실패 → 구현 → 통과”** 입니다. 입력/출력이 흐리면 **테스트가 명세가 아니라 메모**가 됩니다.

### 입력이 명확해야 하는 이유

| 모호한 입력 | 생기는 문제 |
|-------------|-------------|
| “4×4 격자”만 말함 | 빈 칸(0)? null? 문자열? |
| “사용자가 채운다” | 부분 격자인지, 완성 격자인지 |
| “마방진 만들기” | 시드·고정 패턴·임의 해 중 무엇인지 |

**명확한 입력 예(개념):**

- `int[4][4] grid` — 완성 또는 부분(이때 **빈 칸 표현 규약** 필수)
- 또는 **생성 요청**: 인자 없음 / `size=4`만 허용 등

→ 테스트는 **`validate(grid)`**, **`magicConstant(4)`**, **`generate4x4()`** 처럼 **함수 시그니처 = 계약**이 됩니다.

### 출력이 명확해야 하는 이유

| 모호한 출력 | 생기는 문제 |
|-------------|-------------|
| “맞는 것 같음” | true/false 없이 통과 판정 불가 |
| “격자 출력” | 검증만 할 때도 격자를 바꿔 반환하는지 |
| “오류” | 어느 줄이 틀렸는지 없으면 디버깅·학습 목적(Why #2) 상실 |

**명확한 출력 예(개념):**

- `boolean isValid`
- 또는 `ValidationResult { ok, wrongLines[], magicSum }`
- 생성 시: `int[4][4]` **且** 테스트에서 즉시 `assertTrue(validate(result))`

**Why #1과의 연결:**  
“완성”을 성공으로 두려면, 출력은 최소 **`isValid == true`이거나, 생성 결과가 검증을 통과`**해야 합니다. 그렇지 않으면 Why #1의 “완성 = 증거”가 프로그램 안에서 **재현되지 않습니다**.

**Why #2와의 연결:**  
검증 자동화·오류 방지는 **입력 격자 + 출력 판정**이 한 쌍으로 고정될 때만 의미가 있습니다.

---

## Why #3 종합: TDD를 선택하는 이유 (핵심 발견)

> **4×4 마방진은 규칙·불변량·판정이 분명한 문제이므로, TDD로 “통제할 것(검증·집합·합·경계)”과 “불변 조건(특히 M=34, 완성 시 전 줄 합 일치)”을 테스트로 먼저 고정하고, 입력/출력을 계약으로 명확히 해 두면, 구현 방식(손법·탐색·하드코딩)을 바꿔도 “마방진이냐”라는 의미가 흔들리지 않기 때문이다.**

```text
        [TDD가 고정하는 것]
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
 통제      불변 조건   I/O 계약
(검증·회귀) (M=34 등)  (격자 in → bool/격자 out)
    │         │         │
    └─────────┴─────────┘
              │
    Why #1: 완성 = validate 통과
    Why #2: 자동 검증·반복·실수 방지
```

### TDD로 설계할 때 드러나는 다음 긴장 (STEP 5로 넘길 씨앗)

1. **검증만 먼저 TDD** vs **생성까지 한 번에 TDD** — 범위 선택  
2. **알려진 해 하나를 하드코딩**해도 `validate` 테스트는 통과 — “생성의 어려움”은 별도 테스트 필요  
3. **부분 격자**를 다룰지에 따라 테스트 수·복잡도 급증

# STEP 5 — 진짜 문제 정의

---

## 1. 표면 문제 정의 (잘못된 정의)

> **“4×4 격자에 1부터 16까지 숫자를 넣어서, 행·열·대각선 합이 같은 마방진을 완성하는 프로그램을 만든다.”**

### 왜 잘못된 정의인가

| 문제 | 설명 |
|------|------|
| 목적이 한 문장에 과다 | “만든다”에 생성·출력·학습·시연·과제 제출이 모두 섞임 |
| 성공 기준 불명확 | “완성”이 최종 격자인지, 과정인지, 화면에 보이는지 불분명 |
| 역할 미분리 | 규칙을 **만족하는지 판단**하는 일과 **새 배치를 얻는** 일이 구분되지 않음 |
| 입력·상태 무시 | 빈 칸, 잘못된 숫자, 부분 채움, 사용자 입력 등이 정의 밖 |
| 실패·경계 없음 | 맞지 않을 때·아직 채우는 중일 때·규모가 다를 때를 다루지 않음 |
| 검증 가치 누락 | “맞다”는 **누가, 어떤 규칙으로, 무엇을 근거로** 말하는지 없음 |

이 정의는 **퍼즐 완료 행위**를 프로그램 전체의 정체성으로 두어, STEP 2~4에서 드러난 **반복·자동 검증·오류 방지·규칙 기반 사고·TDD적 통제**와 맞지 않습니다.

---

## 2. 개선된 문제 정의 (정확한 정의)

> **고정된 규모(4×4)와 고정된 숫자 집합(1~16, 각각 정확히 한 번) 위에서, “모든 행·모든 열·두 대각선의 합이 하나의 동일한 값과 같다”는 조건을 만족하는 배치인지를 일관되게 판정하고, 필요할 때만 그 조건을 만족하는 배치를 제시할 수 있어야 한다. 판정 기준은 명시적이며 재현 가능하고, 판정 결과는 동일 입력에 대해 항상 같아야 한다.**

### 정의에 포함되는 것

| 요소 | 내용 |
|------|------|
| **대상** | 4×4 배치, 칸마다 1~16 중 하나(중복·누락 없음이 완성 상태의 요건) |
| **규칙** | 4개 행, 4개 열, 2개 대각선 — 각각의 합이 서로 같을 것 |
| **핵심 능력 1** | 주어진 배치가 규칙을 만족하는지 **판정** |
| **핵심 능력 2 (선택)** | 규칙을 만족하는 배치를 **제시** — 단, 제시한 것은 반드시 판정을 통과해야 함 |
| **품질 요구** | 같은 배치에 대해 판정 결과가 흔들리지 않음; 판정 규칙이 문서·검사와 일치 |

### 정의에서 의도적으로 비우는 것

- 어떤 방식으로 배치를 얻는지
- 해가 하나인지 여러 개인지 전부 나열하는지
- 화면·입력 방식·사용자 경험
- 4×4 이외의 일반화

→ **“무엇이 옳은 배치인가”** 와 **“그것을 믿을 수 있게 판정하는가”** 가 중심이고, **“어떻게 채우는가”** 는 부가 목표입니다.

## 3. 이 문제의 핵심 Invariant

### 최종 상태 (완성 배치)

| ID | Invariant | 의미 |
|----|-----------|------|
| **I1** | 집합 불변 | 16개 값의 집합은 정확히 {1, 2, …, 16} |
| **I2** | 합 불변 | 네 행, 네 열, 두 대각선 — 각각의 합이 모두 동일 |
| **I3** | 마법 상수 | 위에서 말한 공통 합은 34 (4×4, 1~16에서 유도되는 고정값) |
| **I4** | 판정 일치 | “규칙 만족”으로 인정된 배치는 I1~I3을 모두 만족 |

### 시스템·계약 (프로그램이 지켜야 할 불변)

| ID | Invariant | 의미 |
|----|-----------|------|
| **I5** | 판정 결정성 | 동일한 배치 입력 → 동일한 만족/불만족 결과 |
| **I6** | 제시-판정 일치 | (제시 기능이 있을 때) 제시된 배치는 판정에서 반드시 만족 |
| **I7** | 실패 명시 | 불만족일 때 “조용한 성공” 없음 — 틀림은 판정 가능한 상태로 남음 |

### 구분해야 할 비-Invariant (혼동 방지)

| 항목 | 성격 |
|------|------|
| 특정 대칭·패턴·구성 방법 | 문제 정의에 넣지 않는 한 불변 아님 |
| “한 줄만 맞음” | 최종 불변 아님; 중간 학습 상태일 수 있음 |
| “대각선 제외” | 다른 문제 정의이며, 본 문제의 불변 아님 |

# 4x4 Magic Square — TDD 설계 문서 생성 프롬프트

**용도:** STEP 6 설계 문서를 생성하기 위한 Cursor Agent 실행용 프롬프트  
**선행 조건:** `Report/01.problem-definition.md` 완료  
**출력 대상:** `Report/02.design.md`

---

## 실행용 프롬프트

```
당신은 TDD 설계 전문가입니다.
나는 4x4 Magic Square 검증 시스템을 TDD 방식으로 구현하려 합니다.

아래 문서를 먼저 읽고, 그 내용을 설계의 유일한 근거로 삼으십시오.
참조 문서: Report/01.problem-definition.md

이 문서에 정의된 7개의 불변 조건을 기반으로, STEP 6 설계 문서를 작성하십시오.

--------------------------------------------
STEP 6-1 — 기술 스택 결정
--------------------------------------------
다음 항목을 결정하고, 각 선택의 근거를 한 문장으로 기술하십시오.

1. 구현 언어
2. 테스트 프레임워크
3. 프로젝트 디렉토리 구조 (파일 트리 형태로 표현)

제약:
- 근거는 불변 조건 또는 TDD 사이클과 연결되어야 합니다.
- 도구의 장점 나열 금지. "왜 이 프로젝트에 적합한가"만 기술하십시오.

--------------------------------------------
STEP 6-2 — 인터페이스 명세
--------------------------------------------
검증 시스템의 공개 인터페이스를 정의하십시오.

요구사항:
1. 외부에서 호출하는 진입점 함수는 정확히 1개입니다.
2. 입력 타입과 출력 타입을 명시하십시오.
3. 출력은 단순 참/거짓이 아닌, 어떤 조건이 실패했는지를 포함해야 합니다.
4. 내부 구현 함수(각 불변 조건 검증 함수)의 시그니처도 모두 정의하십시오.
5. 함수 본문은 작성하지 마십시오. 시그니처와 역할 설명만 기술하십시오.

--------------------------------------------
STEP 6-3 — 테스트 케이스 명세
--------------------------------------------
7개의 불변 조건 각각에 대해 테스트 케이스를 명세하십시오.

각 불변 조건마다 다음 3가지를 기술하십시오:

[정상 케이스 - GREEN]
- 해당 조건을 만족하는 입력의 특성

[실패 케이스 - RED]
- 해당 조건만 위반하고 나머지는 만족하는 입력의 특성
- 예상 출력 (어떤 조건 항목이 실패로 보고되어야 하는가)

[경계 케이스]
- 해당 조건과 관련된 경계 입력의 특성 (빈 입력, 최솟값/최댓값, null 등)

제약:
- 실제 코드(assert, expect 등) 작성 금지.
- 테스트 케이스의 의도와 입출력 특성만 자연어로 기술하십시오.
- 7개의 불변 조건을 모두 다루어야 합니다.

--------------------------------------------
STEP 6-4 — TDD 실행 순서 계획
--------------------------------------------
7개의 불변 조건을 어떤 순서로 구현할지 결정하고, 그 이유를 기술하십시오.

요구사항:
1. 구현 순서를 1번부터 7번까지 나열하십시오.
2. 각 순서의 이유를 "이전 조건과의 의존성" 또는 "독립성"으로 설명하십시오.
3. 각 불변 조건에 해당하는 브랜치 이름을 함께 명시하십시오.
   - 브랜치 형식: red/{name}, green/{name}, refactoring/{name}

--------------------------------------------
STEP 6-5 — 설계 자기 검증
--------------------------------------------
작성한 설계가 다음 기준을 모두 충족하는지 확인하십시오.

체크리스트:
- [ ] 7개의 불변 조건 각각에 대응하는 검증 함수가 존재하는가
- [ ] 진입점 함수가 7개 조건을 모두 위임하는가
- [ ] RED 테스트 케이스가 7개 조건 모두에 존재하는가
- [ ] 경계 케이스가 각 조건마다 1개 이상 정의되어 있는가
- [ ] 인터페이스 명세에 구현 코드가 포함되어 있지 않은가
- [ ] TDD 실행 순서가 의존성 순서와 일치하는가

체크리스트 항목마다 충족/미충족 여부와 그 근거를 한 줄로 기술하십시오.

--------------------------------------------
공통 제약 (전체 STEP에 적용)
--------------------------------------------
- 구현 코드 작성 금지.
- 알고리즘 설명 금지.
- 각 STEP은 독립된 섹션으로 구분하십시오.
- 구조화된 Markdown으로 작성하십시오.
- 작성 완료 후 Report/02.design.md 파일로 저장하십시오.
```

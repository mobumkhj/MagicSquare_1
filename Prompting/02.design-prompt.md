```

        └── feature/{invariant-name} ← 각 불변 조건별 TDD 사이클
```

### feature 브랜치 목록 (불변 조건 단위)

| 브랜치 이름 | 대상 불변 조건 |

### TDD 사이클 흐름

```
developer
  └─ feature/invariant-grid-size
        │
        ├── [RED]      test: 4x4가 아닌 격자 거부 테스트 (실패하는 테스트 작성)
        ├── [GREEN]    impl: 격자 크기 검증 최소 구현 (테스트 통과)
        ├── [GREEN]    impl: 엣지 케이스 추가 통과
        ├── [REFACTOR] refactor: 검증 메서드 분리
        └── [REFACTOR] refactor: 네이밍 정리
              │
              └──▶ developer 로 PR/merge (GREEN 상태 보장 확인 후)
                        │
                        └──▶ master 로 merge (마일스톤 완료 시)
```

### 커밋 메시지 컨벤션

```
[RED]      test: 격자 크기가 5x4일 때 유효하지 않음을 검증
[GREEN]    impl: 행 수 != 4 조건으로 격자 크기 검증 통과
[REFACTOR] refactor: isValidGridSize() 메서드로 추출
```

### 핵심 규칙 (3가지)

1. **`master`에는 RED 커밋이 절대 들어가지 않는다** — 항상 모든 테스트가 GREEN인 상태만
2. **feature 브랜치는 불변 조건 하나 단위로 짧게 유지** — 오래 살아있는 브랜치 없이
3. **`developer` → `feature` 분기, `feature` → `developer` 머지** — master는 마일스톤 완료 시에만

## `developer`에서 `red` / `green` / `refactoring` 브랜치 전략 깊이 검토

> 참고로 `refactory`는 영어에 존재하지 않는 단어입니다. `refactoring` 또는 `refactor`가 올바른 표현입니다.

### 전략의 구조 파악

제안하신 구조는 다음과 같이 이해됩니다.

```
developer
  ├── spec
  ├── red          ← TDD RED 단계 전담 브랜치
  ├── green        ← TDD GREEN 단계 전담 브랜치
  └── refactoring  ← TDD REFACTORING 단계 전담 브랜치
```

이 세 브랜치가 **공유 장기 브랜치(shared long-lived branches)** 로 운영된다는 가정 아래 검토합니다.

---

### 핵심 문제: 이 브랜치들을 어떻게 "이동"하는가

TDD 사이클은 반복(iteration)입니다. 불변 조건이 7개라면 이 사이클이 7번 반복됩니다.

#### 방식 A — 공유 브랜치 재사용

| 관점 | 평가 |
|---|---|
| 단순함 | 브랜치가 3개로 고정되어 명확함 |
| 히스토리 | 사이클 1의 `red` 기록이 사라짐 |
| 학습 가치 | TDD 사이클의 누적 과정을 기록으로 남기지 못함 |
| 병렬 불가 | 한 번에 하나의 불변 조건만 처리 가능 |

#### 방식 B — 불변 조건별로 새 브랜치 생성

```
red/invariant-grid-size
green/invariant-grid-size
refactoring/invariant-grid-size
```

| 관점 | 평가 |
|---|---|
| 히스토리 | 각 불변 조건의 모든 단계가 완전히 보존됨 |
| 가시성 | 브랜치 이름만으로 "무엇의 어떤 단계"인지 즉시 파악 가능 |
| 브랜치 수 | 불변 조건 7개 × 3단계 = **21개** 브랜치 |
| 복잡도 | 브랜치 관리 부담이 커짐 |

## 브랜치 전략 선택지 요약

### 선택지 1 — 기능(불변 조건) 단위 브랜치 + 커밋 컨벤션

```
developer
  └── feature/invariant-{name}
        ├── [RED]      커밋
        ├── [GREEN]    커밋
        └── [REFACTORING] 커밋
```

TDD 단계는 브랜치가 아닌 **커밋 메시지 접두사**로 표현합니다.

| 장점 | 단점 |
|---|---|
| 브랜치 수가 적고 단순함 | 현재 어느 TDD 단계인지 브랜치 이름으로 알 수 없음 |

---

### 선택지 2 — TDD 단계 공유 브랜치 (단순형)

```
developer
  ├── spec
  ├── red
  ├── green
  └── refactoring
```

사이클마다 이 브랜치들을 **재사용**합니다.

| 장점 | 단점 |
|---|---|
| 브랜치 수 고정, 구조 단순 | 이전 사이클 히스토리가 사라짐 |
| 현재 TDD 단계가 브랜치 이름으로 명확 | 불변 조건 간 맥락이 혼재 |

---

### 선택지 3 — TDD 단계 × 불변 조건 조합 브랜치 (절충안)

```
developer
  ├── spec
  ├── red/invariant-{name}
  ├── green/invariant-{name}
  └── refactoring/invariant-{name}
```

| 장점 | 단점 |
|---|---|
| TDD 단계 + 불변 조건 맥락이 브랜치 이름에 모두 표현됨 | 브랜치 수: 불변 조건 7개 × 3단계 = **21개** |
| 각 사이클의 히스토리가 완전히 보존됨 | 브랜치 관리 부담 증가 |

---

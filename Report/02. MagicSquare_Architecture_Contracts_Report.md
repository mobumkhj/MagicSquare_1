### 고정 입출력 계약 (외부)

**입력**

| 규칙 | 내용 |
|------|------|
| 형식 | `4×4 int[][]` (0 = 빈칸) |
| 빈칸 | 정확히 **2개** |
| 값 범위 | `0` 또는 `1~16` |
| 중복 | `0` 제외 중복 **금지** |

**출력**

| 규칙 | 내용 |
|------|------|
| 형식 | `int[6]` |
| 좌표 | **1-index** |
| 배열 | `[r1, c1, n1, r2, c2, n2]` |
| 숫자 | `n1`, `n2` = 두 누락 숫자 |
| 순서 | `(작은수→첫 빈칸, 큰수→둘째 빈칸)` 조합이 마방진이면 `[small, large]`; 아니면 반대 |
| 빈칸 순서 | **행 우선(row-major)** 스캔 시 첫 `0` = 첫 빈칸, 둘째 `0` = 둘째 빈칸 |

# 1) Logic Layer (Domain Layer) 설계

## 1.1 도메인 개념

| 구분 | 이름 | 책임 (SRP) |
|------|------|------------|
| **Value Object** | `Grid4x4` | 4×4 정수 격자 보유. 도메인 연산용 불변 스냅샷(0=빈칸 허용). 크기·인덱스 접근만 담당. |
| **Value Object** | `CellPosition` | 1-index `(row, col)`. 동등성·정렬 키 제공. |
| **Value Object** | `MagicConstant` | 4×4·1~16 기준 **34** 단일 값. 합 비교의 기준. |
| **Value Object** | `FilledGrid4x4` | 16칸 모두 1~16, 중복 없음. **완전 격자**만 표현. |
| **Entity** | `PartialPuzzle` | `Grid4x4` + UI 계약 통과 후 도메인 솔버 입력. |
| **Domain Service** | `EmptyCellLocator` | 빈칸(0) 좌표를 **행 우선**으로 2개 찾아 `first`, `second` 고정. |
| **Domain Service** | `MissingNumberFinder` | 1~16 중 격자에 없는 수 2개를 오름차순 `[small, large]` 반환. |
| **Domain Service** | `MagicSquareJudge` | 완전 격자가 I1~I3 만족 여부를 `true`/`false`로 판정. |
| **Domain Service** | `TwoAssignmentTrial` | `(small→firstBlank, large→secondBlank)` / 반대 배치 2회 생성 후 판정. |
| **Domain Service** | `SolutionOrderResolver` | 출력 규칙에 맞는 `[n1, n2]` 순서와 좌표 쌍 결정. |
| **Application Service** | `SolveTwoBlanksUseCase` | 위 서비스 조합: `PartialPuzzle` → `Solution6` 또는 도메인 실패. |

**의존 방향:** `SolveTwoBlanksUseCase` → Domain Services → Value Objects. UI·Repository는 참조하지 않음.

## 1.2 도메인 불변조건(Invariants)

| ID | 불변조건 | 검증 가능 조건 |
|----|----------|----------------|
| **I1** | 완전 격자의 16값 = {1,…,16} 각 1회 | multiset equals 1..16 |
| **I2** | 4행·4열·주대각·부대각 합이 모두 동일 | 10개 합 pairwise equal |
| **I3** | I2의 공통 합 = **34** | 10개 합 모두 34 |
| **I4** | “마방진이다” ⇔ I1∧I2∧I3 | `MagicSquareJudge` 단일 진실원 |
| **I5** | 동일 `FilledGrid4x4` → 동일 판정 | 순수 함수 |
| **I6** | 빈칸 순서: **행 우선** 첫 0 = `first`, 둘째 0 = `second` | 좌표 결정 규칙 고정 |
| **I7** | 해 없음 시 **침묵 성공 금지** | 예외 또는 `Failure` 반환; 길이 6 배열 금지 |
| **I8** | 출력 숫자 순서: `(small→first, large→second)`가 마방진이면 `[n1,n2]=[small,large]`; 아니면 반대 | 2회 시도 규칙 |

### 엣지 규칙 (검증 가능)

| 상황 | 도메인 결과 |
|------|-------------|
| 시도 A·B 중 **정확히 1개** 마방진 | 해당 배치로 `Solution6` 생성 |
| **0개** 마방진 | `NoValidCompletion` 실패 |
| **2개** 마방진 (이론상) | **시도 A**(small→first) 우선; A 성공이면 A 순서, A 실패·B 성공이면 B 순서 |

## 1.3 핵심 유스케이스(도메인 관점)

| UC-ID | 이름 | 단계 | 산출 |
|-------|------|------|------|
| **UC-D1** | 빈칸 찾기 | `Grid4x4` 스캔(행1→4, 열1→4) | `first`, `second` : `CellPosition` |
| **UC-D2** | 누락 숫자 찾기 | 1~16 − 격자의 비0 값 | `small`, `large` |
| **UC-D3** | 마방진 판정 | `FilledGrid4x4` → 10선 합 | `boolean` |

| **UC-D5** | 솔루션 순서 결정 | A 판정 → 성공 시 `[small,large]`; 실패 시 B 판정 | `Solution6` |
| **UC-D6** | 통합 솔브 | UC-D1~D5 | `int[6]` 도메인 모델 |

## 1.4 Domain API(내부 계약)

> 실패 표현: `Result` 또는 checked 예외 중 **프로젝트 1종**만 선택. 테스트 계약은 동일.

### `EmptyCellLocator.locate(Grid4x4 grid)`

| 항목 | 내용 |
|------|------|
| 입력 | `grid` — 4×4, 0=빈칸 |
| 출력 | `(CellPosition first, CellPosition second)` |
| 전제 | 빈칸 정확히 2개(UI 검증 후) |
| 실패 | 빈칸 ≠ 2 → `InvalidPuzzleState` |

### `MissingNumberFinder.findMissing(Grid4x4 grid)`

| 항목 | 내용 |
|------|------|
| 입력 | `grid` |
| 출력 | `(int small, int large)`, `small < large`, 둘 다 1~16 |
| 전제 | 비0 값 중복 없음, 비0 개수 = 14 |
| 실패 | 누락 수 ≠ 2 → `InvalidPuzzleState` |

### `MagicSquareJudge.isMagic(FilledGrid4x4 grid)`

| 항목 | 내용 |
|------|------|
| 입력 | 완전 격자 |
| 출력 | `true` iff I1∧I2∧I3 |
| 실패 | 없음(불리언만) |

### `TwoAssignmentTrial.buildCandidates(PartialPuzzle puzzle)`

| 항목 | 내용 |
|------|------|
| 출력 | `TrialA`, `TrialB` 각 `FilledGrid4x4` |
| 실패 | 없음 |

### `SolutionOrderResolver.resolve(TrialA, TrialB, positions, small, large)`

| 항목 | 내용 |
|------|------|
| 출력 | `Solution6 { r1, c1, n1, r2, c2, n2 }` — 좌표 1-index |
| 실패 | A·B 모두 false → `NoValidCompletion` |
| 규칙 | I8·엣지(2해) 표 준수 |

### `SolveTwoBlanksUseCase.execute(PartialPuzzle puzzle)`

| 항목 | 내용 |
|------|------|
| 출력 | `Solution6` |
| 실패 | `NoValidCompletion` |

**`Solution6` 불변:** `length==6`, `r1,c1,r2,c2 ∈ [1,4]`, `n1,n2 ∈ [1,16]`, `n1≠n2`, `(r1,c1)≠(r2,c2)`.

# 2) Screen Layer (UI Layer) 설계 (Boundary Layer)

## 2.1 사용자/호출자 관점 시나리오

```
Caller → PuzzleBoundary.submit(matrix)
       → InputValidator.validate(matrix)
       → [invalid] ErrorResponse (code, message)
       → [valid]   SolveTwoBlanksUseCase.execute(...)
       → [domain fail] ErrorResponse DOMAIN_NO_SOLUTION
       → [ok] int[6] SuccessResponse
```

| 단계 | 행위 |
|------|------|
| 1 | 호출자가 `int[4][4]` 제출 |
| 2 | Boundary가 구조·값·빈칸·중복 검증 |
| 3 | 실패 → 표준 `ErrorResponse` |
| 4 | 성공 → Domain 실행 (UI 테스트 시 Mock) |
| 5 | 성공 → `int[6]` 또는 Domain 실패 코드 |

**Dual-Track:** UI 계약 테스트는 **Domain Mock**; Domain 테스트는 **Boundary 미참조**.

## 2.2 UI 계약(외부 계약)

### Input schema

| 필드 | 타입 | 규칙 |
|------|------|------|
| `matrix` | `int[4][4]` | 필수 |
| `matrix[r][c]` | `int` | `0` 또는 `1..16` |
| 빈칸 | — | `0` 개수 **== 2** |
| 중복 | — | `0` 제외 값 중복 **0건** |
| 크기 | — | 행=4, 열=4 (각 내부 행 길이 4) |

### Output schema (성공)

| 필드 | 타입 | 규칙 |
|------|------|------|
| `result` | `int[6]` | `[r1, c1, n1, r2, c2, n2]` |
| 좌표 | — | 각 `1..4` |
| 숫자 | — | `n1`, `n2` ∈ 1..16, 서로 다름 |
| 의미 | — | `(r1,c1)`=행우선 첫 빈칸, `(r2,c2)`=둘째; `n1,n2`는 Domain I8 |

### Error schema

| 필드 | 타입 | 규칙 |
|------|------|------|
| `code` | `string` | 아래 enum 고정 |
| `message` | `string` | 아래 문구와 **바이트 동일** |
| `result` | — | 없음 |

| code | 권장 HTTP | message (고정 문자열) |
|------|-----------|-------------------------|
| `INVALID_SIZE` | 400 | `Matrix must be 4x4.` |
| `INVALID_EMPTY_COUNT` | 400 | `Exactly 2 empty cells (0) are required.` |
| `INVALID_VALUE_RANGE` | 400 | `Each cell must be 0 or an integer from 1 to 16.` |
| `INVALID_DUPLICATE` | 400 | `Non-zero values must not duplicate.` |
| `DOMAIN_NO_SOLUTION` | 422 | `No valid magic square completion exists.` |
| `STORAGE_NOT_FOUND` | 404 | `Stored puzzle not found.` |
| `INTERNAL_ERROR` | 500 | `Unexpected error.` |

**인덱스:** Boundary 외부 입력은 **0-index** 배열; 출력 좌표만 **1-index**.

| 학습 목표 | 인터페이스 분리 | 직렬화·예외 경계 |

### 추천: **B (JSON 파일)**

1. 테스트 데이터를 저장소에 커밋 가능  
2. `CorruptFormat`·`NotFound`를 실파일로 RED 가능  
3. 통합 테스트 속도는 **InMemory Fake**로 보완  

**파일 레이아웃:**

```
data/puzzles/{id}.json   → { "matrix": [[...], [...], [...], [...]] }
data/results/{id}.json   → { "result": [r1, c1, n1, r2, c2, n2] }
```

# 4) Integration & Verification

## 4.1 통합 경로 정의

```
Caller
  → PuzzleBoundary (UI)
      → InputValidator
      → SolveTwoBlanksUseCase (Application)
          → EmptyCellLocator, MissingNumberFinder,
             TwoAssignmentTrial, MagicSquareJudge,
             SolutionOrderResolver (Domain)
      → MatrixRepository (Data, optional)
```

| 경로 | 흐름 |
|------|------|
| **P0 (필수)** | Boundary → Validator → UseCase → Domain → `int[6]` |
| **P1 (선택)** | loadPuzzle → P0 → saveResult |
| **의존성** | Domain ← Application ← Boundary; Data는 Domain에 의존하지 않음 |

## 4.2 통합 테스트 시나리오

### 정상 (2건 이상)

| ID | 시나리오 | 검증 |
|----|----------|------|
| IT-OK1 | 유효 2빈칸 행렬 E2E | `result` = Domain 골든과 일치 |
| IT-OK2 | save puzzle → load → solve → saveResult → loadResult | 행렬·결과 동일 |

### 실패 (3건 이상)

| ID | 시나리오 | 검증 |
|----|----------|------|

| IT-F2 | 해 없는 2빈칸 | `DOMAIN_NO_SOLUTION` |
| IT-F3 | 없는 id 로드 | `STORAGE_NOT_FOUND` |

# PRD — Magic Square 4x4 TDD Practice

## 1. Executive Summary

이 프로젝트는 4×4 Magic Square를 “풀어내는 알고리즘”이 아니라, **불변식 기반 사고**와 **입력/출력 계약 고정**, **Boundary/Domain 분리(Dual-Track UI + Logic TDD)**, **RED-GREEN-REFACTOR 흐름**, **Concept-to-Code Traceability**를 구현 전에 확정하는 TDD 훈련 프로젝트이다. 고정된 입력(4×4, 0=blank, blank=2, 값 범위/중복 규칙)과 고정된 출력(`int[6]`, 1-index 좌표, small-first→reverse 시도 규칙)을 기준으로, Boundary는 계약 검증과 오류 표준화를 담당하고 Domain은 순수 규칙(불변식) 판정을 담당한다.

- 이 프로젝트가 훈련하려는 핵심 역량:
  - 불변식 사고(“항상 참이어야 하는 규칙”을 테스트 가능한 문장으로 고정)
  - 입력/출력 계약(스키마/포맷/오류)을 구현 전에 고정
  - Dual-Track TDD(Track A: Boundary 계약, Track B: Domain 불변식)
  - 설계 → 테스트 → 구현 → 리팩토링 흐름(RED 확인 없이는 구현 금지)
  - Concept → Rule → Use Case → Contract → Test → Component 추적성 유지

## 2. Background

본 프로젝트의 출발점은 “마방진을 완성하는 프로그램”이 아니라, **규칙·불변 조건·판정 계약**을 명확히 하고 이를 **반복 가능하게 검증**하는 데 있다. 4×4(1~16) 문제는 규모가 작아 규칙을 전부 열거하고 검증하기에 적합하며, 이 과정에서 **검증 자동화, 오류 방지, 책임 분리, 회귀 보호**를 학습 목표로 삼는다. 구현을 먼저 시작하면 성공 기준이 흔들리고(무엇이 ‘맞음’인가), 테스트가 뒤늦게 붙으면서 계약이 계속 바뀌는 문제가 발생하므로, PRD가 “구현 전 기준 문서”로서 계약과 불변식을 먼저 고정한다.

---

## 3. Problem Statement

이 프로젝트의 문제는 “마방진을 만든다”가 아니라 다음을 만족하는 시스템을 구현하는 것이다.

- **검증 가능한 불변식 조건**을 명확히 정의한다.
- 고정된 입력/출력 계약을 기준으로, 동일 입력에 대해 동일 결과를 반환한다(결정성).
- Boundary와 Domain을 분리해, Boundary는 계약(입력/출력/오류)을 책임지고 Domain은 순수 규칙(불변식 판정/조합 규칙)을 책임진다.

입력/출력 계약은 이 시스템의 “정의 그 자체”이며, 계약이 흔들리면 테스트가 명세 역할을 수행할 수 없으므로 반드시 고정한다.

## 4. Why Now / Why Chain

지금 이 프로젝트를 수행하는 이유는 다음의 반복되는 학습 실패 패턴을 끊기 위함이다.

- 학습자가 겪는 문제(관찰 가능한 형태):
  - 구현부터 시작하여 “성공/실패 기준”이 구현에 종속된다.
  - 테스트 기준이 불명확하여 테스트가 요구사항이 아니라 사후 검증으로 전락한다.
  - Boundary(입력 검증/오류/포맷)와 Domain(규칙 판정)이 섞여 변경 영향 범위가 확대된다.
  - 리팩토링 이후 출력 포맷/오류 문구/규칙이 변형되어 회귀가 발생한다.

- Why chain(요약):
  - **왜 프로그램으로?** 반복 가능성/검증 자동화/오류 방지/규칙 기반 사고 훈련을 위해
  - **왜 TDD로?** “무엇이 통제되어야 하는가(불변식·계약)”를 구현 전에 고정해 의미가 흔들리지 않게 하기 위해
  - **왜 Dual-Track?** 계약(UI/Boundary)과 규칙(Domain/Logic)을 분리하여 테스트/구현 진행을 병렬화하고 책임 혼합을 차단하기 위해

## 10. Functional Requirements

### FR-01 Input Verification (Boundary)

- Description: 입력 행렬이 계약을 만족하는지 검증하고, 실패 시 표준 오류를 반환한다.
- Layer: Boundary
- Input: `int[4][4]`로 취급 가능한 정수 행렬
- Processing Rules:
  - 행 수는 4, 각 행의 열 수는 4여야 한다.
  - 값은 `0` 또는 `1..16`이어야 한다.
  - `0`의 개수는 정확히 2개여야 한다.
  - `0`을 제외한 값은 중복될 수 없다.
- Output: 성공 시 Domain 호출 / 실패 시 `ErrorResponse { code, message }`
- Acceptance Criteria:
  - 크기 위반 → `INVALID_SIZE`
  - 빈칸 개수 위반 → `INVALID_EMPTY_COUNT`
  - 범위 위반 → `INVALID_VALUE_RANGE`
  - 중복 위반 → `INVALID_DUPLICATE`
  - 위 4종 실패에서는 Domain solver를 호출하지 않는다.
- Error / Exception Policy: §13
- Related Business Rules: BR-01 ~ BR-04
- Related Test Direction: Track A
- Component Candidate: `InputValidator`, `PuzzleBoundary`

### FR-02 Blank Coordinate Discovery (Domain)

- Description: row-major 순서로 0(빈칸) 2개의 좌표를 첫 번째/두 번째로 결정한다.
- Layer: Domain
- Input: Boundary 검증 통과 4×4 행렬
- Processing Rules: row-major(행 1→4, 열 1→4) 탐색, 첫 0 = `firstBlank`, 둘째 0 = `secondBlank`
- Output: `(r1,c1)`, `(r2,c2)` 내부 표현(1-index)
- Acceptance Criteria: 동일 입력 → 동일 좌표, `firstBlank`는 row-major 첫 0
- Related Business Rules: BR-05
- Component Candidate: `BlankFinder`

### FR-03 Missing Number Discovery (Domain)

- Description: 1..16 중 입력에 없는 숫자 2개를 오름차순(small, large)으로 반환한다.
- Layer: Domain
- Acceptance Criteria: 누락 2개, `small < large`
- Related Business Rules: BR-06, BR-07
- Component Candidate: `MissingNumberFinder`

### FR-04 Magic Square Validation (Domain)

- Description: 완성된 4×4 격자가 마방진 불변식을 만족하는지 판정한다.
- Layer: Domain
- Processing Rules: Magic Constant 34, 4행·4열·2대각선 합 = 34, 값 집합 {1..16} 각 1회
- Output: `true/false`
- Acceptance Criteria: I1~I3 부합 시 true, 위반 시 false, 결정적
- Related Business Rules: BR-08 ~ BR-11
- Component Candidate: `MagicSquareValidator`

### FR-05 Two-Combination Solver and Result Formatting

- Description: 두 빈칸에 누락 숫자 2개를 두 방식으로 대입하여 마방진이 되는 경우 `int[6]` 반환
- Layer: Domain + Boundary
- Processing Rules:
  - Attempt 1: `firstBlank←small`, `secondBlank←large` → 판정
  - Attempt 2: Attempt 1 실패 시 반대 조합 → 판정
  - Attempt 1 성공 → `n1=small`, `n2=large`
  - Attempt 2 성공 → `n1=large`, `n2=small`
  - 둘 다 실패 → `DOMAIN_NO_SOLUTION`
- Output: `int[6] = [r1,c1,n1,r2,c2,n2]`
- Acceptance Criteria: 길이 6, 좌표 1-index, Attempt별 n 순서 규칙
- Related Business Rules: BR-05 ~ BR-15
- Component Candidate: `Solver`, `ResultFormatter`, `PuzzleBoundary`

## 11. Business Rules / Domain Rules

| ID | Rule |
|----|------|
| BR-01 | 입력은 4×4여야 한다 |
| BR-02 | 값 0의 개수는 정확히 2개여야 한다 |
| BR-03 | 각 셀 값은 0 또는 1..16이어야 한다 |
| BR-04 | 0을 제외한 값은 중복될 수 없다 |
| BR-05 | row-major 첫 0이 첫 번째 빈칸이다 |
| BR-06 | 누락 숫자는 정확히 2개여야 한다 |
| BR-07 | 누락 숫자는 오름차순 (small, large)이다 |
| BR-08 | Magic Constant는 34이다 |
| BR-09 | 4개 행의 합은 각각 34이다 |
| BR-10 | 4개 열의 합은 각각 34이다 |
| BR-11 | 2개 대각선의 합은 각각 34이다 |
| BR-12 | 출력 좌표는 1-index(1..4)이다 |
| BR-13 | 출력은 `int[6] = [r1,c1,n1,r2,c2,n2]`이다 |
| BR-14 | Attempt 1: small→첫 빈칸, large→둘째 빈칸 |
| BR-15 | Attempt 1 실패 시 Attempt 2: large→첫 빈칸, small→둘째 빈칸 |

## 12. Input / Output Contract

### Input Contract

| Field | Type | Rule | Invalid Example | Error Code |
|-------|------|------|-----------------|------------|
| matrix | int[4][4] | 4×4 | 3×4, jagged | `INVALID_SIZE` |
| cell value | int | 0 또는 1..16 | -1, 17 | `INVALID_VALUE_RANGE` |
| blanks | count(0) | 정확히 2 | 0, 1, 3개 | `INVALID_EMPTY_COUNT` |
| duplicates | non-zero | 0 제외 중복 금지 | 5가 2회 | `INVALID_DUPLICATE` |

### Output Contract (Success)

| Field | Type | Rule |
|-------|------|------|
| result | int[6] | `[r1,c1,n1,r2,c2,n2]` |
| coordinates | int | 1..4 (1-index) |
| n1, n2 | int | 1..16, `n1≠n2` |
| ordering | rule | Attempt1 성공: n1=small; Attempt2 성공: n1=large |

### Output Contract (Error)

| Field | Type | Rule |
|-------|------|------|
| code | string | §13 enum |
| message | string | 코드별 고정 문자열 |
| result | absent | 오류 응답에 result 없음 |

## 13. Error / Failure Policy

| Condition | Error Code | Message | Layer | Domain 호출 |
|-----------|------------|---------|-------|-------------|
| 4×4 아님 | `INVALID_SIZE` | `Matrix must be 4x4.` | Boundary | **미호출** |
| blank ≠ 2 | `INVALID_EMPTY_COUNT` | `Exactly 2 empty cells (0) are required.` | Boundary | **미호출** |
| 범위 위반 | `INVALID_VALUE_RANGE` | `Each cell must be 0 or an integer from 1 to 16.` | Boundary | **미호출** |
| 중복 | `INVALID_DUPLICATE` | `Non-zero values must not duplicate.` | Boundary | **미호출** |
| 두 조합 실패 | `DOMAIN_NO_SOLUTION` | `No valid magic square completion exists.` | Domain→Boundary | **호출(유효 입력)** |

- Domain 내부 실패 표현: `NoValidCompletion`
- Boundary 변환: `DOMAIN_NO_SOLUTION`

> **Decision Needed:** Domain 실패를 예외 vs Result/Failure 객체 중 1종으로 선택한다.

## 17. Architecture Overview

- **Boundary:** 입력 검증, 오류 응답, Control 호출, 결과 반환
- **Control:** Boundary↔Domain 오케스트레이션
- **Domain:** 순수 마방진 로직, 불변식 검증

**의존 방향:** Boundary → Control → Domain

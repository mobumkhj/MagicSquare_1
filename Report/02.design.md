## STEP 6-1 — 기술 스택 결정

### 구현 언어: Python

> 4x4 격자의 행·열·대각선을 불변 조건 단위로 분리하여 검증하는 구조가 리스트 연산으로 간결하게 표현되며, RED→GREEN 전환 속도가 빠르고 TDD 사이클에 집중할 수 있다.

### 테스트 프레임워크: pytest

> 불변 조건 7개를 독립된 테스트 함수로 분리하고, 어느 조건이 실패했는지를 항목 단위로 보고하는 구조가 "완성의 불가분성" 조건 검증과 자연스럽게 대응된다.

## STEP 6-2 — 인터페이스 명세

### 타입 정의

| 타입 이름 | 구조 | 설명 |
|---|---|---|
| `Grid` | `list[list[int]]` | 4x4 정수 배열 |
| `ConditionResult` | `{ name: str, passed: bool, reason: str \| None }` | 단일 불변 조건 판정 결과 |
| `ValidationResult` | `{ is_valid: bool, failed_conditions: list[ConditionResult] }` | 전체 검증 결과 |

- `name`: 불변 조건 이름 (예: "격자 크기", "행 합")
- `passed`: 해당 조건 통과 여부
- `reason`: 실패 시 사유, 통과 시 `None`
- `is_valid`: 7개 조건 모두 통과 시에만 `True`
- `failed_conditions`: 실패한 조건들의 목록 (통과 시 빈 리스트)

---

### 진입점 함수 (공개, 1개)

```
validate(grid: Grid) -> ValidationResult
```

- **입력:** `Grid` — 검증 대상 4x4 정수 배열
- **출력:** `ValidationResult` — 전체 유효 여부 + 실패 조건 목록
- **역할:** 내부 검증 함수 7개를 순서대로 위임하고, 각 결과를 통합하여 최종 판정을 반환한다.

---

### 내부 검증 함수 (비공개, 7개)

| 함수 시그니처 | 역할 |
|---|---|
| `_check_grid_size(grid: Grid) -> ConditionResult` | 격자가 정확히 4행 4열인지 확인한다 |
| `_check_number_set(grid: Grid) -> ConditionResult` | 1부터 16까지의 정수가 모두 존재하는지 확인한다 |
| `_check_no_duplicate(grid: Grid) -> ConditionResult` | 어떤 숫자도 두 번 이상 등장하지 않는지 확인한다 |
| `_check_row_sums(grid: Grid) -> ConditionResult` | 4개 행 각각의 합이 34인지 확인한다 |
| `_check_col_sums(grid: Grid) -> ConditionResult` | 4개 열 각각의 합이 34인지 확인한다 |
| `_check_diag_sums(grid: Grid) -> ConditionResult` | 주 대각선과 반 대각선 각각의 합이 34인지 확인한다 |
| `_check_completeness(results: list[ConditionResult]) -> ConditionResult` | 앞의 6개 조건이 모두 통과된 경우에만 완성 상태로 판정한다 |

> 진입점 `validate()`는 위 7개 함수를 순서대로 호출하여 결과 목록을 수집하고, 실패 항목 여부로 `is_valid`를 결정한다.

## STEP 6-3 — 테스트 케이스 명세

---

### 불변 조건 1 — 격자 크기 (4x4)

**[GREEN — 정상 케이스]**  
4개의 행과 각 행마다 4개의 열로 구성된 격자가 주어질 때, `_check_grid_size`는 통과 결과를 반환한다.

**[RED — 실패 케이스]**  
- 3x4 격자 (행이 3개인 경우) → 격자 크기 조건 실패 보고
- 4x3 격자 (열이 3개인 경우) → 격자 크기 조건 실패 보고
- 5x5 격자 → 격자 크기 조건 실패 보고
- 예상 출력: `name="격자 크기", passed=False, reason="행 또는 열의 수가 4가 아님"`

**[경계 케이스]**  
- 빈 리스트(`[]`) → 실패 (0x0으로 인식)
- 행은 4개이나 각 행의 길이가 서로 다른 경우 → 실패
- 4x4 구조이나 중첩 리스트가 아닌 단일 리스트로 전달될 경우 → 실패

---

### 불변 조건 2 — 숫자 집합 (1~16 완전 포함)

**[GREEN — 정상 케이스]**  
격자에 1부터 16까지의 정수가 모두 포함된 경우 (중복, 합 여부와 무관하게 집합 조건만 확인), `_check_number_set`은 통과 결과를 반환한다.

**[RED — 실패 케이스]**  
- 16 대신 17이 들어간 격자 → 숫자 집합 조건 실패
- 0이 포함된 격자 → 숫자 집합 조건 실패
- 특정 숫자(예: 7)가 빠진 격자 → 숫자 집합 조건 실패
- 예상 출력: `name="숫자 집합", passed=False, reason="허용되지 않는 숫자 포함 또는 필수 숫자 누락"`

**[경계 케이스]**  
- 1과 16이 모두 존재하는 경우 → 통과 (최솟값·최댓값 경계 확인)
- 음수가 포함된 경우 → 실패

---

### 불변 조건 3 — 중복 금지

**[GREEN — 정상 케이스]**  
1~16이 각각 정확히 한 번씩 등장하는 격자가 주어질 때, `_check_no_duplicate`는 통과 결과를 반환한다.

**[RED — 실패 케이스]**  
- 숫자 5가 두 번 등장하고 숫자 11이 빠진 격자 → 중복 조건 실패
- 예상 출력: `name="중복 금지", passed=False, reason="중복 숫자 존재: [5]"`

**[경계 케이스]**  
- 16개 칸 모두 동일한 숫자(예: 전부 1)인 격자 → 실패
- 중복이 단 1개만 있는 경우 → 실패 (하나의 중복도 허용되지 않음)

---

### 불변 조건 4 — 행 합 = 34

**[GREEN — 정상 케이스]**  
4개 행 각각의 합이 모두 34인 격자가 주어질 때, `_check_row_sums`는 통과 결과를 반환한다.

**[RED — 실패 케이스]**  
- 1번 행의 합이 33이고 나머지 행은 34인 격자 → 행 합 조건 실패
- 예상 출력: `name="행 합", passed=False, reason="1번 행의 합이 34가 아님 (실제: 33)"`

**[경계 케이스]**  
- 행 합은 모두 34이지만 숫자 집합이 1~16이 아닌 격자 → 행 합 조건은 통과, 숫자 집합 조건 실패 (조건 독립성 확인)

---

### 불변 조건 5 — 열 합 = 34

**[GREEN — 정상 케이스]**  
4개 열 각각의 합이 모두 34인 격자가 주어질 때, `_check_col_sums`는 통과 결과를 반환한다.

**[RED — 실패 케이스]**  
- 3번 열의 합만 35인 격자 → 열 합 조건 실패
- 예상 출력: `name="열 합", passed=False, reason="3번 열의 합이 34가 아님 (실제: 35)"`

**[경계 케이스]**  
- 행 합은 모두 34이나 열 합이 34가 아닌 격자 → 열 합 조건 실패, 행 합 조건 통과 (방향별 독립성 확인)

---

### 불변 조건 6 — 대각선 합 = 34

**[GREEN — 정상 케이스]**  
주 대각선과 반 대각선 각각의 합이 모두 34인 격자가 주어질 때, `_check_diag_sums`는 통과 결과를 반환한다.

**[RED — 실패 케이스]**  
- 주 대각선 합은 34이나 반 대각선 합이 30인 격자 → 대각선 합 조건 실패
- 예상 출력: `name="대각선 합", passed=False, reason="반 대각선의 합이 34가 아님 (실제: 30)"`

**[경계 케이스]**  
- 행·열 합은 모두 34이나 대각선 합이 34가 아닌 격자 → 대각선 합 조건 실패 (행/열과 독립적으로 검증됨을 확인)

---

### 불변 조건 7 — 완성의 불가분성

**[GREEN — 정상 케이스]**  
앞의 6개 불변 조건을 모두 만족하는 완전한 마방진 격자가 주어질 때, `validate()`는 `is_valid=True`, `failed_conditions=[]`를 반환한다.

**[RED — 실패 케이스]**  
- 행 합 조건만 실패하고 나머지 5개 조건은 통과하는 격자 → `is_valid=False` 반환
- 예상 출력: `is_valid=False, failed_conditions=[{name="행 합", passed=False, ...}]`

**[경계 케이스]**  
- 6개 조건 중 5개만 통과한 경우 → 여전히 `is_valid=False` (단 1개의 실패로 전체가 무효임을 확인)

## STEP 6-4 — TDD 실행 순서 계획

브랜치 전략: **선택지 2** — `red`, `green`, `refactoring` 공유 브랜치를 사이클마다 재사용

| 순서 | 불변 조건 | 의존성 근거 | 브랜치 (사이클) |
|:---:|---|---|---|
| 1 | 격자 크기 | 모든 조건의 전제. 4x4가 아니면 다른 어떤 검증도 의미 없으므로 독립적으로 가장 먼저 구현 | `red` → `green` → `refactoring` (사이클 1) |
| 2 | 숫자 집합 | 격자 크기 확인 이후, 어떤 숫자가 허용되는지를 정의. 중복·합 조건 모두의 전제가 됨 | `red` → `green` → `refactoring` (사이클 2) |
| 3 | 중복 금지 | 숫자 집합 조건과 연관되나 "같은 숫자 반복"이라는 별도 위반 패턴을 독립적으로 검증 | `red` → `green` → `refactoring` (사이클 3) |
| 4 | 행 합 | 합 조건 중 방향성이 가장 직관적이며, 열·대각선 합과 독립적으로 구현 가능 | `red` → `green` → `refactoring` (사이클 4) |
| 5 | 열 합 | 행 합 구조를 방향만 바꿔 재적용. 행 합 구현 이후 자연스러운 연속 | `red` → `green` → `refactoring` (사이클 5) |
| 6 | 대각선 합 | 행·열 합 이후 마지막 방향 조건. 주 대각선과 반 대각선을 각각 독립적으로 확인 | `red` → `green` → `refactoring` (사이클 6) |
| 7 | 완성의 불가분성 | 앞의 6개 조건 전체에 의존하는 통합 조건. 반드시 마지막에 구현 | `red` → `green` → `refactoring` (사이클 7) |

### 브랜치 흐름 (사이클당)

```
developer
  └─▶ red      (실패하는 테스트 작성)
        └─▶ green        (테스트를 통과하는 최소 구현)
              └─▶ refactoring   (코드 구조 개선)
                    └─▶ developer (머지 후 브랜치 리셋, 다음 사이클 시작)
```

## STEP 6-5 — 설계 자기 검증

| 항목 | 결과 | 근거 |
|---|:---:|---|
| 7개의 불변 조건 각각에 대응하는 검증 함수가 존재하는가 | ✅ 충족 | STEP 6-2에서 `_check_grid_size` 외 7개 함수 시그니처 모두 명세됨 |
| 진입점 함수가 7개 조건을 모두 위임하는가 | ✅ 충족 | `validate()`가 7개 내부 함수를 순서대로 위임하고 결과를 통합하는 구조로 명세됨 |
| RED 테스트 케이스가 7개 조건 모두에 존재하는가 | ✅ 충족 | STEP 6-3에서 7개 조건 모두에 실패 케이스와 예상 출력이 기술됨 |
| 경계 케이스가 각 조건마다 1개 이상 정의되어 있는가 | ✅ 충족 | 7개 조건 모두에 경계 케이스 최소 2개 이상 기술됨 |
| 인터페이스 명세에 구현 코드가 포함되어 있지 않은가 | ✅ 충족 | 함수 시그니처와 역할 설명만 기술, 함수 본문 없음 |
| TDD 실행 순서가 의존성 순서와 일치하는가 | ✅ 충족 | 격자 크기(전제) → 숫자 집합 → 중복 → 행/열/대각선 합(독립) → 완성(통합) 순서로 의존성 방향과 일치 |

> **6개 항목 모두 충족.** 설계 문서는 STEP 7 구현 착수의 기준 명세로 사용 가능하다.

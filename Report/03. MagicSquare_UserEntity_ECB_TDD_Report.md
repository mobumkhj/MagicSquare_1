## 1. 목적

ECB 아키텍처에서 `entity` 레이어에 위치하는 `User` 도메인 엔티티를 구현하고, pytest + AAA 패턴으로 해당 엔티티의 핵심 불변 조건을 테스트로 고정한다.

---

## 2. 적용 규칙(.cursorrules)

- Python 3.14+
- PEP8, 최대 라인 길이 88
- 모든 함수 파라미터/반환값 타입힌트
- Google 스타일 docstring (public 메서드)
- pytest + AAA (Arrange-Act-Assert)
- ECB 의존성 방향: boundary → control → entity (entity는 순수 도메인)
- TDD: RED → GREEN → REFACTOR
- 금지: `print(...)` 디버깅, 테스트 약화, `except:` 단독 등

---

## 3. ECB 관점 설계 요약

### 3.1 엔티티 책임

`User`는 도메인 데이터(`user_id`, `display_name`)와 그에 대한 규칙(표시 이름의 공백/길이 제약)을 가진다.

### 3.2 엔티티 순수성

`src/entity/user.py`는 다음을 포함하지 않는다.

- I/O (CLI/API/파일/네트워크)
- 프레임워크/인프라 의존
- 출력 기반 디버깅(`print`)

---

## 4. 불변 조건(도메인 규칙)

- `display_name`은 공백만으로 구성될 수 없다.
- `display_name`은 좌우 공백이 제거된(정규화된) 형태로 저장된다.
- `display_name` 길이는 `MAX_DISPLAY_NAME_LEN` 이하이다.
- `rename()`는 동일 `user_id`를 유지하면서 새 인스턴스를 반환한다(불변 객체).

---

## 5. 테스트(AAA) 설계

테스트 파일: `tests/entity/test_user.py`

- Arrange: `User` import 및 입력 준비
- Act: `User.create()` 또는 `rename()`
- Assert: 생성된 값의 타입/동치, 예외 발생, 불변성(새 인스턴스) 확인

테스트 케이스:

- ID가 없으면 UUID를 생성한다.
- 공백 이름은 거부한다.
- 최대 길이를 초과하는 이름은 거부한다.
- rename은 ID를 유지하고 새 인스턴스를 반환한다.

---

## 6. 구현 요약

파일: `src/entity/user.py`

- `@dataclass(frozen=True, slots=True)`로 불변 엔티티 모델링
- `__post_init__`에서 입력 정규화/검증 수행
- `create()` 팩토리 메서드로 ID 생성 책임을 엔티티 내부로 캡슐화
- `MAX_DISPLAY_NAME_LEN`는 `ClassVar[int]`로 선언해 dataclass 필드에서 제외
